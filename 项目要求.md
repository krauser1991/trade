## 一、文档目录要求

后续在本项目中生成的所有正式文档，都必须在正文前提供目录。

目录要求：

- Markdown 文档必须包含 `## 目录`；
- 目录位置放在标题、日期、文档属性之后，正文第一章之前；
- 目录至少覆盖一级章节；长文档需要覆盖到二级章节；
- PDF 文档如果由 Markdown 转换生成，必须保留目录页或目录段落；
- 复盘、交易策略、产业链梳理、价值投资报告、会议纪要、交易体系文档都适用；
- 除非用户明确要求极短便签，否则不能省略目录。

这条规则是项目默认要求，不需要用户每次单独提醒。

## 二、文档生成要求

正式文档默认结构：

```text
# 标题

生成日期：
文档属性：

## 目录

- [一、章节](#一章节)
- [二、章节](#二章节)

## 一、章节

正文
```

交易类文档需要尽量包含：

- 市场状态；
- 资金路径；
- 核心锚点；
- 持仓或候选标的处理；
- 仓位计划；
- 风险条件；
- 次日或盘中执行清单。

## 三、PDF 生成要求

生成 PDF 时必须检查：

- PDF 是否包含目录；
- 中文字体是否正常；
- 表格和正文是否重叠；
- 页数和文件路径是否明确；
- 重要文档需要渲染抽查页面后再交付。

## 四、Git 自动提交与推送要求

本项目已经由 Git 托管。后续在本项目中每次生成或更新正式文件后，默认自动完成 GitHub 推送，不需要用户再次确认。

执行要求：

- 正式 Markdown、PDF、Excalidraw、CSV、资产文件、项目要求文件等，生成完成后需要纳入提交；
- PDF 文档应在生成、基础校验、必要页面抽查完成后，再执行提交和推送；
- 临时脚本、PDF 渲染检查图、缓存文件、过程性中间文件，除非用户明确要求保留，否则不作为正式交付文件提交；
- 提交前必须查看 `git status`，确认只提交本次任务相关文件；
- 提交信息使用简洁中文，说明本次新增或更新的文档主题；
- 提交成功后自动执行 `git push` 推送到 GitHub；
- 最终回复中说明提交与推送结果，并给出关键文件路径。

这条规则是项目默认要求，不需要用户每次单独提醒。
